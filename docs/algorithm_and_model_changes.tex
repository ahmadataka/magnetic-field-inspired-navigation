\subsection*{What Was Implemented}

The current cleanroom reference includes:

\begin{itemize}[leftmargin=*]
  \item 2D double-integrator dynamics
  \item 2D circular obstacle abstraction
  \item goal-relaxation gain logic inspired by the ROS implementation
  \item magnetic-field-inspired obstacle avoidance based on the ROS \texttt{field=2} branch
  \item additive total command
\end{itemize}

\begin{align}
a &= a_{\text{goal}} + a_{\text{obs}} \\
v_{k+1} &= v_k + a_k \Delta t \\
p_{k+1} &= p_k + v_{k+1}\Delta t
\end{align}

\subsection*{Goal Relaxation Implemented}

Let
\begin{itemize}[leftmargin=*]
  \item $p \in \mathbb{R}^2$ be robot position
  \item $v \in \mathbb{R}^2$ be robot velocity
  \item $g \in \mathbb{R}^2$ be goal position
  \item $r_g = g - p$ be the goal vector
  \item $r_o$ be the closest obstacle vector
  \item $d_g = \lVert r_g \rVert$
  \item $d_o = \lVert r_o \rVert$
\end{itemize}

The relaxation scale used in the cleanroom implementation is

\begin{equation}
k_{\text{relax}}
=
\left(1 - e^{-d_o/1.5}\right)
\cdot
w_{\angle}
\cdot
w_{\text{prog}}
\cdot
w_{\text{exit}}
\end{equation}

where

\begin{equation}
w_{\angle}
=
1 - \cos \theta
=
1 - \frac{r_g^\top r_o}{\lVert r_g \rVert \lVert r_o \rVert}
\end{equation}

\begin{equation}
w_{\text{prog}}
=
\begin{cases}
1, & d_g \le d_{g,0} \\
\exp\left(-\dfrac{d_g - d_{g,0}}{0.1}\right), & d_g > d_{g,0}
\end{cases}
\end{equation}

and

\begin{equation}
w_{\text{exit}}
=
\exp\left(-\dfrac{d_g - d_{g,\min}}{0.1}\right)
\end{equation}

with:

\begin{itemize}[leftmargin=*]
  \item $d_{g,0}$: initial distance to the goal
  \item $d_{g,\min}$: smallest distance to the goal reached so far
\end{itemize}

This follows the structure of the ROS code around the \texttt{goal\_relax == 1} block.

\subsection*{Goal Controller Implemented}

For the far field, the current cleanroom implementation uses a simplified
speed-and-turn controller inspired by the ROS geometric block:

\begin{equation}
a_{\text{goal}} = a_{\text{attr}} + a_{\text{turn}}
\end{equation}

\begin{equation}
a_{\text{attr}}
=
-\left(k_p\,k_{\text{relax}}\right)
\left(\lVert v \rVert - v_{\max}\right)\hat{s}
\end{equation}

where $\hat{s}$ is:

\begin{itemize}[leftmargin=*]
  \item the normalized velocity direction if $\lVert v \rVert > 0$
  \item otherwise the normalized goal direction
\end{itemize}

The turning term is

\begin{equation}
a_{\text{turn}}
=
k_{\omega}\,k_{\text{relax}}\,\lVert v \rVert\,\hat{n}\,(\hat{n}^\top \hat{g})
\end{equation}

where:

\begin{itemize}[leftmargin=*]
  \item $\hat{g} = r_g / \lVert r_g \rVert$
  \item $\hat{n}$ is the left-perpendicular of the current heading
\end{itemize}

Near the goal, the controller switches to a PD-like attractive form:

\begin{equation}
a_{\text{goal}}
=
k_{p,\text{goal}} r_g - k_d v
\end{equation}

\subsection*{Magnetic-Field-Inspired Obstacle Term Implemented}

The cleanroom port uses the \texttt{field == 2} structure from the old code.

Let
\begin{itemize}[leftmargin=*]
  \item $r_o$ be the closest vector from robot to obstacle surface
  \item $d = \lVert r_o \rVert$
  \item $\hat{l}_a = v / \lVert v \rVert$ be the normalized robot motion direction
  \item $r_{\text{from obs}} = -r_o$
  \item $\hat{r} = r_{\text{from obs}} / \lVert r_{\text{from obs}} \rVert$
\end{itemize}

The obstacle current direction is the tangential projection of the robot motion:

\begin{equation}
\hat{t}
=
\hat{l}_a - (\hat{l}_a^\top \hat{r})\hat{r}
\end{equation}

If this becomes degenerate, the implementation falls back to a perpendicular
direction.

The 2D magnetic field surrogate is modeled as

\begin{equation}
B \propto \frac{\lVert v \rVert}{d}\,\hat{t}_{\perp}
\end{equation}

and the magnetic avoidance term is

\begin{equation}
F_{\text{mag}}
=
k_m \hat{l}_{a,\perp}(\hat{l}_{a,\perp}^\top B)
\end{equation}

This is a 2D analogue of the nested cross-product structure in the ROS code:

\begin{align}
B &= [\hat{t}]_{\times} v / d \\
F &= k_m [\hat{l}_a]_{\times} B
\end{align}

\subsection*{Added Repulsive Correction Implemented}

The cleanroom implementation also keeps the old additional repulsive term near
the obstacle:

\begin{equation}
F_{\text{rep}}
=
-k_r \left(\frac{1}{d} - \frac{1}{d_b}\right)\frac{r_o}{d}
\quad \text{for } d \le d_b
\end{equation}

Then it is projected to remove the component along the motion direction:

\begin{equation}
F_{\text{rep}}^{\perp}
=
F_{\text{rep}} - (F_{\text{rep}}^\top \hat{l}_a)\hat{l}_a
\end{equation}

The final obstacle term is:

\begin{equation}
a_{\text{obs}} = F_{\text{mag}} + F_{\text{rep}}^{\perp}
\end{equation}

\subsection*{Major Simplifications Relative To The Old ROS Code}

These are intentional and important:

\begin{enumerate}[leftmargin=*]
  \item The current reference is strictly 2D. The ROS code contains both 2D and 3D branches.
  \item Obstacles are currently circles. The ROS stack computes obstacle geometry from point clouds, polygonal segments, and local obstacle surfaces.
  \item The far-field goal controller is structurally inspired by the ROS code, but it is not yet a line-by-line reproduction of the SO(3)-style geometric block.
  \item The magnetic term is mathematically faithful in structure, but the current implementation uses a compact 2D analogue rather than a direct symbolic port of the original 3D cross-product expressions.
  \item The cleanroom model removes all ROS-specific concerns: TF, publishers/subscribers, launch files, point-cloud nodes, timestamps, and platform adapters.
\end{enumerate}

\subsection*{Why This Change Was Made}

The goal of this first cleanroom step is verification by isolation:

\begin{itemize}[leftmargin=*]
  \item keep the algorithm small enough to reason about
  \item remove ROS coupling
  \item make rollout behavior inspectable with CSV and plot outputs
  \item establish a stable place for future refactors and comparisons
\end{itemize}

\subsection*{Required Follow-Up}

Future major changes should update this file, especially if they modify:

\begin{itemize}[leftmargin=*]
  \item the exact goal-relaxation gain
  \item the magnetic obstacle formula
  \item the near-goal switching logic
  \item the state update model
  \item the obstacle abstraction
  \item the benchmark scenario assumptions
\end{itemize}

\section*{Entry 2026-05-22: Structural Refactor Toward The 2022 Paper}

\textbf{Status}

\begin{itemize}[leftmargin=*]
  \item Refactored in \texttt{src/mfinav/double\_integrator.py}
  \item This is a structural alignment step, not yet a full equation-level reconstruction of the 2022 paper
\end{itemize}

\textbf{Motivation}

The 2022 paper describes obstacle avoidance as the combination of two terms:

\begin{equation}
F_o = F_b + F_a
\end{equation}

where:

\begin{itemize}[leftmargin=*]
  \item $F_b$ is the boundary-following field
  \item $F_a$ is the collision-avoidance field
\end{itemize}

The earlier cleanroom prototype merged these ideas into one obstacle term plus
an added repulsive correction. That made the code easy to test, but it hid the
paper's core decomposition.

\textbf{What Changed}

The obstacle logic is now split into explicit components:

\begin{itemize}[leftmargin=*]
  \item \texttt{LocalSensingModel}
  \item \texttt{BoundaryFollowingField}
  \item \texttt{CollisionAvoidanceField}
\end{itemize}

The total control now has the explicit structure

\begin{equation}
a = a_{\text{goal}} + a_b + a_a
\end{equation}

where:

\begin{itemize}[leftmargin=*]
  \item $a_b$ is the boundary-following component
  \item $a_a$ is the collision-avoidance component
\end{itemize}

\textbf{Observation Model}

The refactor introduces a local observation container:

\begin{equation}
\mathcal{O} = \{ r_o,\; d_o \}
\end{equation}

where:

\begin{itemize}[leftmargin=*]
  \item $r_o$ is the closest-obstacle vector
  \item $d_o = \lVert r_o \rVert$
\end{itemize}

This is still generated from an analytic circle model in the current code, but
the new interface is intentionally shaped like a local sensing output so it can
be replaced later by sensor-driven obstacle processing.

\textbf{Boundary-Following Field}

The boundary-following component keeps the same current prototype behavior, but
it is now isolated as its own field:

\begin{equation}
\hat{t} = \hat{l}_a - (\hat{l}_a^\top \hat{r})\hat{r}
\end{equation}

\begin{equation}
B \propto \frac{\lVert v \rVert}{d}\hat{t}_{\perp}
\end{equation}

\begin{equation}
a_b = k_m \hat{l}_{a,\perp}(\hat{l}_{a,\perp}^\top B)
\end{equation}

This is still the compact 2D analogue used previously, but it is now clearly
identified as the boundary-following term.

\textbf{Collision-Avoidance Field}

The repulsive near-obstacle correction is now made explicit as the
collision-avoidance field:

\begin{equation}
a_a =
-k_r \left(\frac{1}{d} - \frac{1}{d_b}\right)\frac{r_o}{d}
\quad \text{for } d \le d_b
\end{equation}

and, when motion direction is defined, it is projected to remove the component
parallel to the current motion:

\begin{equation}
a_a^{\perp} = a_a - (a_a^\top \hat{l}_a)\hat{l}_a
\end{equation}

\textbf{Goal Relaxation Status}

The goal-relaxation logic remains available, but it is now explicitly marked as
a legacy extension from the ROS code rather than a confirmed core part of the
2022 paper implementation.

This is represented by a configuration switch:

\begin{equation}
\texttt{use\_legacy\_goal\_relaxation} \in \{0,1\}
\end{equation}

When disabled, the goal controller uses unit scaling rather than the legacy
relaxation factor.

\textbf{What This Refactor Improves}

\begin{itemize}[leftmargin=*]
  \item The code structure now mirrors the paper's decomposition more clearly.
  \item It becomes easier to replace the current placeholder formulas with
  paper-faithful ones.
  \item It isolates local sensing, boundary following, and collision avoidance
  as separate concerns.
\end{itemize}

\textbf{What Still Does Not Match The 2022 Paper Exactly}

\begin{itemize}[leftmargin=*]
  \item The exact paper equations for $F_b$ and $F_a$ are not yet reconstructed line by line.
  \item The threshold semantics of $r_l$ and $r_{la}$ are not yet implemented as separate paper-faithful activations.
  \item The Section 4.4 closest-point averaging extension is still missing.
  \item The current observation model still comes from an analytic obstacle, not real local sensing.
\end{itemize}

\section*{Entry 2026-05-22: Benchmark Environments and APF Baseline}

\textbf{Status}

\begin{itemize}[leftmargin=*]
  \item Added multi-obstacle environment support
  \item Added convex and non-convex benchmark scenarios
  \item Added a standard artificial potential field (APF) baseline
  \item Added a benchmark runner that saves trajectory plots and metrics
\end{itemize}

\textbf{What Changed}

The simulator is no longer restricted to a single obstacle. A new obstacle
collection abstraction returns the closest obstacle vector among several
primitive obstacles:

\begin{equation}
r_o = \arg\min_{r_i \in \mathcal{R}} \lVert r_i \rVert
\end{equation}

where $\mathcal{R}$ is the set of closest-point vectors returned by all
obstacles in the environment.

This enables more realistic environments while preserving the local-sensing
structure used by the controller.

To make controller comparisons numerically meaningful, the simulator now clips
both commanded acceleration and robot speed:

\begin{equation}
a \leftarrow
\begin{cases}
a, & \lVert a \rVert \le a_{\max} \\
a_{\max}\dfrac{a}{\lVert a \rVert}, & \text{otherwise}
\end{cases}
\end{equation}

\begin{equation}
v \leftarrow
\begin{cases}
v, & \lVert v \rVert \le v_{\max} \\
v_{\max}\dfrac{v}{\lVert v \rVert}, & \text{otherwise}
\end{cases}
\end{equation}

The rollout also terminates on collision, approximated by zero obstacle
clearance in the simplified geometric model.

\textbf{New Scenario Types}

Three benchmark scenarios are now defined:

\begin{itemize}[leftmargin=*]
  \item a single convex obstacle
  \item a cluster of convex obstacles
  \item a non-convex U-shaped obstacle approximated by several circles
\end{itemize}

The non-convex environment is still a geometric approximation, but it is useful
for testing whether the controller exhibits boundary-following behavior beyond
simple single-obstacle deflection.

\textbf{APF Baseline}

A standard artificial potential field baseline is added with the structure

\begin{equation}
a_{\text{APF}} = a_{\text{att}} + a_{\text{rep}}
\end{equation}

where the attractive term is

\begin{equation}
a_{\text{att}} = K_P (p_g - p) - K_D v
\end{equation}

and the repulsive term is activated near obstacles:

\begin{equation}
a_{\text{rep}}
=
-K_R \left(\frac{1}{d} - \frac{1}{d_b}\right)\frac{r_o}{d^2}
\quad \text{for } d \le d_b
\end{equation}

This baseline is not meant to reproduce every APF variant in the literature; it
is meant to provide a simple and interpretable comparison against the current
MFI implementation.

\textbf{Verification Outputs}

The benchmark runner now produces:

\begin{itemize}[leftmargin=*]
  \item trajectory comparison plots
  \item CSV metrics for each scenario and method
\end{itemize}

The current summary metrics are:

\begin{itemize}[leftmargin=*]
  \item success flag
  \item collision flag
  \item number of steps
  \item path length
  \item final goal distance
  \item minimum obstacle clearance
  \item mean speed
\end{itemize}

\textbf{Why This Matters}

This change gives us a repeatable way to evaluate:

\begin{itemize}[leftmargin=*]
  \item whether the current MFI implementation handles more complex geometry
  \item whether it offers any practical benefit over a standard APF baseline
  \item where the current simplified implementation breaks down
\end{itemize}

\section*{Entry 2026-05-22: Paper-Driven Thresholds, Averaging Extension, and Legacy Goal Relaxation Demotion}

\textbf{Status}

\begin{itemize}[leftmargin=*]
  \item Added explicit threshold semantics for boundary following and collision avoidance
  \item Added the Section 4.4 style closest-point averaging extension
  \item Changed legacy goal relaxation from default-on to optional
\end{itemize}

\textbf{Thresholded Obstacle Activation}

The implementation now follows the paper structure more closely by separating
the activation thresholds:

\begin{itemize}[leftmargin=*]
  \item boundary-following field active for $d \le r_l$
  \item collision-avoidance field active for $d \le r_{la}$
\end{itemize}

with default values:

\begin{equation}
r_l = 4.0, \qquad r_{la} = 2.5
\end{equation}

This replaces the previous use of generic proximity constants that did not map
cleanly to the paper's notation.

The implementation now also applies smooth activation weights:

\begin{equation}
\sigma_b =
\mathrm{clip}\left(
\frac{r_l - d}{r_l - r_{la}}, 0, 1
\right)
\end{equation}

\begin{equation}
\sigma_a =
\mathrm{clip}\left(
\frac{r_{la} - d}{r_{la}}, 0, 1
\right)
\end{equation}

so that the effective fields are:

\begin{equation}
a_b \leftarrow \sigma_b a_b
\end{equation}

\begin{equation}
a_a \leftarrow \sigma_a a_a
\end{equation}

This makes the obstacle interaction less abrupt and improved the clustered
obstacle benchmark in practice.

\textbf{Closest-Point Averaging Extension}

To approximate the Section 4.4 extension for non-unique closest points, the
observation model now collects all sensed obstacle vectors whose magnitudes are
within a threshold $\delta_r$ of the minimum sensed distance:

\begin{equation}
\mathcal{R}_{\delta}
=
\{ r_i \mid \lVert r_i \rVert \le r_{\min} + \delta_r \}
\end{equation}

The averaged vector is then

\begin{equation}
\bar{r}
=
\frac{1}{|\mathcal{R}_{\delta}|}
\sum_{r_i \in \mathcal{R}_{\delta}} r_i
\end{equation}

Following the paper's heuristic:

\begin{itemize}[leftmargin=*]
  \item if $\lVert \bar{r} \rVert < \lVert r_o \rVert$, use $\bar{r}$ as the working closest-point vector
  \item otherwise, use the standard closest vector $r_o$
\end{itemize}

This gives the code a practical non-unique closest-point handling mechanism for
concave and sharp-corner approximations.

To better reflect the paper's sensing assumption, the implementation no longer
uses only one exact closest-point vector per obstacle. Each obstacle now
provides multiple local surface samples around the nearest visible region, and
the averaging extension operates over these sensed vectors rather than over a
single analytic point per obstacle.

\textbf{Small-Current Fallback}

The paper notes that the projected artificial current can become very small when
the robot direction is nearly aligned with the robot-to-obstacle vector. To
mitigate this, the implementation now checks the magnitude of the projected
current and falls back to a unit tangent surrogate when:

\begin{equation}
\lVert \hat{t} \rVert < \epsilon
\end{equation}

with default

\begin{equation}
\epsilon = 3 \times 10^{-6}
\end{equation}

\textbf{Legacy Goal Relaxation}

The ROS-derived goal-relaxation term is no longer the default path for the
implementation. It is now explicitly treated as an optional legacy extension:

\begin{equation}
\texttt{use\_legacy\_goal\_relaxation} = 0
\end{equation}

by default.

When enabled, the implementation restores the earlier ROS-inspired scaling
factor. When disabled, the goal controller operates without that legacy gain
modulation.

\textbf{Why This Change Was Made}

These changes address three of the most important mismatches between the
prototype and the 2022 paper:

\begin{itemize}[leftmargin=*]
  \item explicit $r_l$ and $r_{la}$ threshold behavior
  \item handling of non-unique closest points
  \item clearer separation between paper-driven behavior and ROS-legacy behavior
\end{itemize}

\section*{Entry 2026-05-22: Tuned Defaults and Stronger Evaluation Metrics}

\textbf{Status}

\begin{itemize}[leftmargin=*]
  \item Increased the default rollout horizon from 2000 to 6000 steps
  \item Tuned the default interaction and sensing parameters around the better-performing region
  \item Added time-to-goal and path-efficiency evaluation metrics
\end{itemize}

\textbf{Tuned Defaults}

The default parameters are now:

\begin{equation}
r_l = 4.0, \quad r_{la} = 2.5, \quad c_{\text{field}} = 12.0, \quad c_{\perp} = 35.0
\end{equation}

and the local sensing surrogate uses:

\begin{equation}
N_{\text{samples}} = 15, \qquad \theta_{\text{window}} = 1.0, \qquad \delta_r = 0.5
\end{equation}

These values were selected because they materially improved the convex-cluster
benchmark while preserving good behavior in the single-convex case.

With the longer rollout horizon, the current MFI implementation is able to
reach the success region in all three default benchmark scenarios.

\textbf{Additional Evaluation Metrics}

The benchmark harness now reports:

\begin{itemize}[leftmargin=*]
  \item time-to-goal in simulation steps
  \item path efficiency
\end{itemize}

with

\begin{equation}
\text{path efficiency}
=
\frac{\lVert p_f - p_0 \rVert}{L}
\end{equation}

where:

\begin{itemize}[leftmargin=*]
  \item $p_0$ is the initial position
  \item $p_f$ is the realized final position
  \item $L$ is the realized trajectory length
\end{itemize}

This definition ensures the metric remains meaningful even when a method does
not reach the goal.

Time-to-goal is defined as the first step index where:

\begin{equation}
\lVert p - p_g \rVert \le r_{\text{success}}
\end{equation}

If the goal region is never reached, the value is reported as $\infty$.

\section*{Entry 2026-05-22: Polygonal Obstacles With Local Sensing}

\textbf{Status}

\begin{itemize}[leftmargin=*]
  \item Added explicit polygon obstacle support
  \item Kept the same local-sensing and closest-point averaging pipeline
  \item Converted the default benchmark environments to polygonal convex and non-convex obstacles
\end{itemize}

\textbf{Polygon Obstacle Model}

A polygon obstacle is represented by an ordered list of boundary vertices

\begin{equation}
\mathcal{P} = \{ v_1, v_2, \dots, v_N \}
\end{equation}

with edges formed by consecutive vertex pairs. For a robot position $p$, the
closest-point vector is computed by projecting onto each boundary segment and
selecting the shortest resulting vector.

\begin{equation}
r_o = \arg\min_{r_i \in \mathcal{E}} \lVert r_i \rVert
\end{equation}

where $\mathcal{E}$ is the set of candidate vectors to all polygon edges.

\textbf{Local Sensing on Polygon Boundaries}

Instead of using only a single exact closest point, the polygon model generates
multiple local samples along each edge around the nearest visible region. These
sampled boundary vectors are passed into the same observation model already used
for circular obstacles:

\begin{itemize}[leftmargin=*]
  \item find the minimum sensed distance
  \item gather all vectors within the averaging band $\delta_r$
  \item compute the averaged closest-point surrogate
  \item use the averaged vector when it is shorter than the single closest vector
\end{itemize}

This keeps the paper-style local sensing and closest-point averaging intact
while making convex and non-convex geometry easier to define.

\textbf{Why This Matters}

Polygonal obstacles let us represent:

\begin{itemize}[leftmargin=*]
  \item single convex obstacles such as boxes or rotated quadrilaterals
  \item clustered convex environments
  \item explicitly non-convex obstacles such as U-shaped boundaries
\end{itemize}

without approximating every case by a large collection of circles.

\section*{Entry 2026-05-22: Signed Clearance Evaluation and Head-On Boundary Fix}

\textbf{Status}

\begin{itemize}[leftmargin=*]
  \item Replaced unsigned polygon distance checks with signed obstacle clearance
  \item Added explicit safety-violation reporting in addition to hard collision reporting
  \item Changed the polygon boundary-following and repulsion terms so a head-on wall encounter no longer degenerates to zero avoidance command
\end{itemize}

\textbf{Signed Clearance}

For evaluation we now distinguish between:

\begin{itemize}[leftmargin=*]
  \item \emph{distance to the sensed boundary} used by the field law
  \item \emph{signed clearance} used by the evaluator
\end{itemize}

For circles,

\begin{equation}
c(p) = \lVert p - p_c \rVert - r
\end{equation}

For polygons, the evaluator computes the minimum Euclidean distance from the
robot position $p$ to any boundary edge and assigns a negative sign if $p$ lies
inside the polygon:

\begin{equation}
c(p) =
\begin{cases}
- \min\limits_{e \in \partial \mathcal{P}} d(p, e), & p \in \mathcal{P} \\
\min\limits_{e \in \partial \mathcal{P}} d(p, e), & p \notin \mathcal{P}
\end{cases}
\end{equation}

This means:

\begin{itemize}[leftmargin=*]
  \item $c(p) \le 0$ indicates collision / penetration
  \item $0 < c(p) \le c_{\text{safety}}$ indicates an unsafe low-clearance path
\end{itemize}

\textbf{Updated Success Accounting}

The benchmark now reports two success-style quantities:

\begin{equation}
\text{goal\_reached\_once} =
\begin{cases}
1, & \exists k \text{ such that } \lVert p_k - p_g \rVert \le r_{\text{success}} \text{ and } c_{\min} > c_{\text{safety}} \\
0, & \text{otherwise}
\end{cases}
\end{equation}

\begin{equation}
\text{success} =
\begin{cases}
1, & \lVert p_f - p_g \rVert \le r_{\text{success}} \text{ and } c_{\min} > c_{\text{safety}} \\
0, & \text{otherwise}
\end{cases}
\end{equation}

The first quantity captures whether the trajectory ever reached the goal region,
while the second remains the stricter final-state metric.

\textbf{Why The Old Polygon Case Failed}

The earlier 2D boundary-following and repulsion approximation had a degeneracy
for a head-on encounter with a vertical wall:

\begin{itemize}[leftmargin=*]
  \item the tangential term collapsed to zero because the projected field and the projected motion-perpendicular direction became orthogonal
  \item the repulsive term collapsed to zero because the full normal repulsion was projected away along the current motion direction
\end{itemize}

That is why parameter sweeps alone could not fix the U-shaped polygon case.

\textbf{Updated Field Form}

To remove that degeneracy, the boundary-following term now directly uses the
local tangent estimate:

\begin{equation}
a_b = \sigma_b \, c_{\text{field}} \, v_{\text{guide}} \, \frac{\hat{t}}{d}
\end{equation}

where

\begin{equation}
v_{\text{guide}} = \max(\lVert v \rVert, 0.5\,v_{\max})
\end{equation}

and $\hat{t}$ is the normalized local tangent estimate (or a perpendicular
fallback when the projected tangent becomes degenerate).

The collision-avoidance term now keeps its full normal component:

\begin{equation}
a_a =
- \sigma_a \, c_{\perp}
\left(
\frac{1}{d} - \frac{1}{r_{la}}
\right)
\frac{r_o}{d}
\end{equation}

This change is less ``projected'' than the previous 2D approximation, but it is
much more appropriate for polygonal wall encounters because it preserves a real
push-away component.

\section*{Entry 2026-06-18: Closer Reconstruction of Paper PD, Geometric, \(F_b\), and \(F_a\)}

\textbf{Status}

\begin{itemize}[leftmargin=*]
  \item Added separate paper-style PD and paper-style geometric presets
  \item Replaced the earlier paper placeholder \(F_b\) with a closer cross-product-based reconstruction
  \item Replaced the earlier smoothed paper placeholder \(F_a\) with a hard-threshold projected repulsive term
  \item Added memory of the previously proposed obstacle-surface current for the small-current degeneracy case
\end{itemize}

\textbf{Paper-style Goal Modes}

The repository now distinguishes:

\begin{itemize}[leftmargin=*]
  \item \texttt{paper\_pd}: direct attractive PD goal control
  \item \texttt{paper\_geometric}: speed regulation plus steering correction
  \item \texttt{pragmatic}: the cleaner benchmark-oriented hybrid controller used in earlier refactors
\end{itemize}

For \texttt{paper\_pd}, the implemented goal term is:

\begin{equation}
u_g = K_P (p_g - p) - K_D \dot{p}
\end{equation}

For \texttt{paper\_geometric}, the far-field speed regulation remains:

\begin{equation}
u_{\text{attr}} = K_{P,r} (v_{\max} - \lVert v \rVert)\hat{v}
\end{equation}

and the steering term is now treated as a 2D analogue of the old SO(3)-based
rotation controller:

\begin{equation}
u_{\text{steer}} = \omega \, v_{\perp},
\qquad
\omega = K_{\omega}\,\mathrm{angle}(v, p_g-p)
\end{equation}

\textbf{Paper-style Boundary-Following Field}

The previous ``paper mode'' used a smoothed 2D surrogate. The updated
implementation is closer to the old ROS magnetic construction and to the paper
derivation:

\begin{equation}
l_o = l_a - \left(l_a^\top \hat{r}\right)\hat{r}
\end{equation}

\begin{equation}
B = [l_o]_{\times} v / d
\end{equation}

\begin{equation}
F_b = c [l_a]_{\times} B
\end{equation}

The implementation now uses a hard activation at \(d \le r_l\) for paper mode
instead of the earlier smooth activation weight.

\textbf{Small-current degeneracy handling}

When \(\lVert l_o \rVert\) becomes too small, the implementation now reuses the
previously proposed normalized obstacle-surface current instead of immediately
switching to the pragmatic tangent fallback. This follows the paper discussion
more closely than the earlier heuristic.

\textbf{Paper-style Collision-Avoidance Field}

For paper mode, the implementation now applies:

\begin{equation}
F_a =
- c_{\perp}
\left(
\frac{1}{d} - \frac{1}{r_{la}}
\right)
\frac{r_o}{d}
\end{equation}

with a projection that removes the component along the instantaneous robot
motion direction in order to better preserve the paper property that \(F_a\)
does not change speed.

\textbf{Observed outcome}

These changes made the paper-aligned PD controller much more plausible on the
simple and clustered convex polygon scenarios. The paper-aligned geometric mode
also became stable after correcting the steering sign in the 2D analogue.

However, the paper-aligned modes still collide in the non-convex U-shaped
polygon benchmark under the current local polygon sensing abstraction, so the
equation-level reconstruction is improved but still not complete.

\section*{Entry 2026-06-18: Matching Paper Actuation, Goal Relaxation, and Local Sensing Assumptions}

\textbf{Status}

\begin{itemize}[leftmargin=*]
  \item Paper-aligned modes now run without acceleration or speed clipping
  \item Goal relaxation is enabled again for the paper-aligned modes
  \item Local sensing for paper-aligned modes now uses ray-based sensing instead of direct analytic boundary sampling
\end{itemize}

\textbf{Actuation Assumption}

The 2022 paper assumes non-saturatable actuators. Earlier versions of the clean
repository applied norm clipping to both acceleration and speed even in the
paper-style modes. This has now been removed for the paper-aligned presets by
setting:

\begin{equation}
\lVert u \rVert_{\max} = \infty,
\qquad
\lVert v \rVert_{\max} = \infty
\end{equation}

for \texttt{paper\_pd} and \texttt{paper\_geometric}.

\textbf{Goal Relaxation Assumption}

Goal relaxation is now active again in the paper-aligned presets. The current
implementation still uses the inherited ROS-era weighting logic, but the paper
modes no longer bypass goal relaxation entirely.

\textbf{Local Sensing Assumption}

Earlier versions used direct local boundary sampling from known analytic circle
or polygon geometry. The paper-aligned sensing path now uses a synthetic
range-sensor style model based on ray casting:

\begin{itemize}[leftmargin=*]
  \item rays are cast around the robot over a full angular sweep
  \item for each ray, the nearest obstacle intersection within the sensor range is selected
  \item only those currently visible local obstacle points are passed to the closest-point and averaging logic
\end{itemize}

This is still a simulation abstraction, but it matches the paper's
``temporally and spatially local sensing'' assumption much better than the
earlier direct-boundary access.

\textbf{Observed outcome}

These assumption changes improved the paper-aligned controllers substantially:

\begin{itemize}[leftmargin=*]
  \item the paper-aligned PD controller now succeeds in the clustered convex polygon benchmark
  \item it now succeeds in the clustered convex polygon benchmark
  \item the paper-aligned geometric controller also succeeds in the clustered convex polygon benchmark
  \item both paper-aligned modes remain collision-free in the non-convex U-shaped benchmark
  \item the paper-aligned geometric controller approaches the non-convex U-shaped benchmark goal much more closely while maintaining safe clearance
\end{itemize}

However, the paper-aligned PD controller still fails to make substantial
progress in the non-convex U-shaped benchmark and the paper-aligned geometric
controller still stops short of full convergence. This suggests that the
remaining discrepancy is no longer just about actuation, goal relaxation, or
local sensing; it is now more likely tied to the exact controller equations and
the polygon benchmark abstraction itself.

\section*{Entry 2026-06-18: Explicit 2D Derivation Target}

\textbf{Status}

\begin{itemize}[leftmargin=*]
  \item Added a dedicated 2D derivation target document for the paper equations
  \item Fixed a concrete target form for \(l_a\), \(n_o\), \(l_o\), \(l_o^\perp\), \(F_b\), and \(F_a\)
  \item Established the exact 2D target to implement against in the next phase
\end{itemize}

\textbf{Why This Was Needed}

The 2022 paper is written in a general 3D vector-field form, while the current
cleanroom benchmark is planar. That left too much room for interpretation in
the 2D implementation. The new derivation note reduces the paper equations to a
precise 2D target before further controller refactors are attempted.

\textbf{Document Added}

\begin{itemize}[leftmargin=*]
  \item \texttt{docs/paper\_2d\_derivation\_target.tex}
\end{itemize}

\textbf{2D Target Highlights}

The derivation note fixes the following working targets:

\begin{equation}
n_o = -\frac{r_o}{\lVert r_o \rVert}
\end{equation}

\begin{equation}
\tilde{l}_o = (I - n_on_o^\top) l_a
\end{equation}

\begin{equation}
l_o =
\begin{cases}
\dfrac{\tilde{l}_o}{\lVert \tilde{l}_o \rVert}, & \lVert \tilde{l}_o \rVert \ge \epsilon \\
l_{o,\text{prev}}, & \lVert \tilde{l}_o \rVert < \epsilon
\end{cases}
\end{equation}

\begin{equation}
l_o^\perp = -l_o
\end{equation}

Using the planar rotation matrix

\begin{equation}
J =
\begin{bmatrix}
0 & -1 \\
1 & 0
\end{bmatrix},
\end{equation}

the working 2D target for the boundary-following field is

\begin{equation}
F_b^{2D}
=
-c
\frac{l_o \times_2 v}{\lVert r_o \rVert}
J l_a
\end{equation}

and the working 2D target for the collision-avoidance field is

\begin{equation}
F_a^{2D}
=
-c_\perp
\frac{l_o^\perp \times_2 r_o}{\lVert r_o \rVert}
J l_a
\end{equation}

\textbf{Caution}

The derivation note is now the implementation target, not yet the final proof
that the current code exactly matches the paper. In particular, the \(F_a\)
expression still needs to be checked carefully against the paper equations and
the ROS implementation.

\section*{Entry 2026-06-18: Step 2 -- \(F_b\) Matched to the Explicit 2D Target}

\textbf{Status}

\begin{itemize}[leftmargin=*]
  \item Replaced the paper-mode boundary-following field implementation with the direct 2D target formula
  \item Removed the embedded-3D surrogate path for the paper-mode \(F_b\)
  \item Verified that benchmark behavior remained stable after the change
\end{itemize}

\textbf{New paper-mode \(F_b\) implementation target}

The paper-mode boundary-following field is now implemented directly as:

\begin{equation}
F_b^{2D}
=
-c
\frac{l_o \times_2 v}{\lVert r_o \rVert}
J l_a
\end{equation}

where:

\begin{equation}
J =
\begin{bmatrix}
0 & -1 \\
1 & 0
\end{bmatrix}
\end{equation}

and \(l_o\) is the obstacle-surface current obtained from the 2D projection
target defined in \texttt{paper\_2d\_derivation\_target.tex}.

\textbf{Why This Matters}

The earlier implementation used an embedded 3D cross-product calculation that
was consistent in spirit, but still left ambiguity about whether the code
matched the exact intended 2D reduction. The new implementation makes the 2D
magnetic field law explicit and removes that ambiguity for \(F_b\).

\textbf{Observed outcome}

Benchmark behavior remained effectively unchanged after replacing the old
paper-mode \(F_b\) with the direct 2D target formula. This is a good sign
because it suggests the derivation target and the previous cross-product-based
implementation were already closely aligned for \(F_b\).

\section*{Entry 2026-06-18: Step 4 -- \(F_a\) Revised Back to a ROS-Backed Short-Range Projected Repulsion}

\textbf{Status}

\begin{itemize}[leftmargin=*]
  \item Rejected the previously tried magnetic \(l_o^\perp\)-based 2D target for \(F_a\)
  \item Restored the paper-mode collision term to a short-range radial repulsion
  \item Kept the orthogonal projection onto the subspace perpendicular to the motion direction
  \item Updated the 2D derivation note so it no longer claims the failed magnetic target is the working \(F_a\) reduction
\end{itemize}

\textbf{Applied target}

The paper-mode collision-avoidance field is now:

\begin{equation}
\tilde{F}_a
=
-c_\perp
\left(
\frac{1}{\lVert r_o \rVert} - \frac{1}{r_{la}}
\right)
\frac{r_o}{\lVert r_o \rVert}
\end{equation}

\begin{equation}

\end{equation}

\textbf{Why this change was necessary}

\(l_o^\perp\)-magnetic reduction for \(F_a\). The stronger implementation
evidence is:

\begin{itemize}[leftmargin=*]
  \item the magnetic cross-product structure already belongs to the boundary-following term,
  \item the close-range collision term is built from the obstacle vector itself,
  \item the result is then projected so the final command remains orthogonal to the motion direction.
\end{itemize}

\textbf{Interpretation}

This makes the clean Python implementation more faithful to the old ROS
behavior and removes a target that was contradicted by both benchmarks and code
audit. It does \emph{not} yet prove that the paper's printed \(F_a\) equation
has been reconstructed exactly, but it is a materially better and better-tested
target than the failed magnetic alternative.

\textbf{Observed outcome}

After this revision:

\begin{itemize}[leftmargin=*]
  \item both paper-aligned modes recovered clean success in the convex polygon benchmarks,
  \item the non-convex U-shaped case returned to collision-free behavior,
  \item the remaining gap is now goal convergence in the non-convex benchmark, not immediate collision.

\begin{itemize}[leftmargin=*]
  \item Added a dedicated \texttt{make\_paper\_geometric\_3d\_config()} preset
  \item Extended the 3D benchmark and interactive HTML viewer to compare \texttt{MFI-PD}, \texttt{MFI-Geometric}, and \texttt{APF}
  \item Reworked the 3D geometric goal controller so it no longer blows up in cluttered scenes
  \item Re-ran the 2D benchmark suite to confirm the new 3D geometric work did not change 2D behavior
\end{itemize}

\textbf{Main controller change}

The first 3D geometric attempt copied the spirit of the 2D turning term too
literally and was numerically unstable. The revised 3D geometric logic now uses:

\begin{itemize}[leftmargin=*]
  \item geometric-style far-field guidance outside the obstacle interaction zone,
  \item PD-style goal attraction once the sensed obstacle distance is within $r_l$,
  \item a 3D lateral turning direction given by the component of the goal direction orthogonal to the current velocity direction,
  \item an angle-weighted turning command rather than a permanently normalized lateral push.
\end{itemize}

\textbf{Current benchmark outcome}

With the stabilized 3D geometric variant:

\begin{itemize}[leftmargin=*]
  \item \texttt{single\_sphere}: success
  \item \texttt{double\_sphere\_gap}: success
  \item \texttt{sphere\_cluster}: success
  \item \texttt{archway}: success
  \item \texttt{offset\_cavity}: still fails by collision
\end{itemize}

\textbf{Interpretation}

\begin{itemize}[leftmargin=*]
  \item the 3D geometric benchmark path is now stable enough to keep in the default comparison,
  \item it is meaningfully stronger than the first unstable 3D geometric draft,
  \item but it is still not as robust as the 3D paper-style PD variant in the offset cavity case.
\end{itemize}

\section*{Entry 2026-06-19: More Faithful 3D Geometric Steering via Skew-Symmetric Structure}

\textbf{Status}

\begin{itemize}[leftmargin=*]
  \item Removed the temporary ``switch to PD inside obstacle range'' idea from the 3D geometric path
  \item Replaced it with a more faithful skew-symmetric steering construction
  \item Re-ran both the 3D and 2D benchmark suites after the change
\end{itemize}

\textbf{Main change}

For the far-field 3D geometric branch, the steering command now uses the nested
skew-symmetric structure built from the current motion direction \(l_a\) and the
goal direction \(l_g\):

\begin{equation}
F_{g,\text{turn}}^{3D}
=
-k_{\text{geom}}
\,[l_a]_{\times}
\left([l_a]_{\times} l_g\right)
\end{equation}

which is equivalent to the lateral projection

\begin{equation}
F_{g,\text{turn}}^{3D}
=
k_{\text{geom}}
\left(I - l_a l_a^\top\right) l_g
\end{equation}

up to the chosen sign convention inside the controller implementation.

This keeps the geometric steering term inside the same cross-product /
skew-symmetric family as the paper's 3D vector formulation, instead of falling
back to a PD branch during obstacle interaction.

\textbf{Current benchmark outcome}

With this more faithful 3D geometric benchmark variant:

\begin{itemize}[leftmargin=*]
  \item \texttt{single\_sphere}: success
  \item \texttt{double\_sphere\_gap}: success
  \item \texttt{sphere\_cluster}: success
  \item \texttt{archway}: success
  \item \texttt{offset\_cavity}: still fails
\end{itemize}

\textbf{Interpretation}

\begin{itemize}[leftmargin=*]
  \item this version is more faithful to the paper's 3D control structure than the temporary hybridized 3D geometric experiment,
  \item it preserves the current 2D benchmark behavior,
  \item and it gives a usable 3D geometric comparison path even though the offset cavity case remains unsolved.
\end{itemize}

\section*{Entry 2026-06-19: 3D Extruded Polygon Obstacles for Convex and Non-Convex Benchmarks}

\textbf{Status}

\begin{itemize}[leftmargin=*]
  \item Added \texttt{PrismObstacle} as an extruded-polygon 3D obstacle model
  \item Extended the 3D benchmark suite with convex and non-convex prism environments
  \item Updated the interactive HTML viewer so prism obstacles appear as 3D wireframe solids
\end{itemize}

\textbf{Model choice}

Instead of introducing a general triangle mesh pipeline, the 3D environment now
uses a simpler and more reusable representation:

\begin{itemize}[leftmargin=*]
  \item a planar polygon footprint in \((x,y)\),
  \item extruded between \(\,z_{\min}\,\) and \(\,z_{\max}\,\).
\end{itemize}

This preserves the spirit of the earlier 2D polygon benchmarks while keeping
the obstacle interface compatible with the local closest-point sensing model.

\textbf{New benchmark environments}

The 3D default benchmark suite now additionally includes:

\begin{itemize}[leftmargin=*]
  \item \texttt{convex\_prism\_block}: a single convex rectangular prism crossing the direct path
  \item \texttt{nonconvex\_prism\_u}: a non-convex U-shaped prism whose opening truly faces the robot
\end{itemize}

\textbf{Current benchmark outcome}

With the current controllers:

\begin{itemize}[leftmargin=*]
  \item \texttt{paper\_pd\_3d} succeeds on both prism environments
  \item \texttt{paper\_geometric\_3d} succeeds on both prism environments
  \item \texttt{apf\_3d} succeeds on the convex prism block but still fails on the non-convex U-prism case
\end{itemize}

\textbf{Interpretation}

\begin{itemize}[leftmargin=*]
  \item the 3D benchmark suite is no longer restricted to sphere-only environments,
  \item convex and non-convex obstacle structure can now be studied directly in 3D using polygon-derived solids,
  \item and the prism model is a better bridge toward future robot and environment extensions than sphere-only approximations.
\end{itemize}

\section*{Entry 2026-06-19: Added Legacy Comparison Navigators}

\textbf{What changed}

\begin{itemize}[leftmargin=*]
  \item Added \texttt{HaddadinNavigator} in \texttt{src/mfinav/navigators/baselines.py}
  \item Added \texttt{SabattiniNavigator} in \texttt{src/mfinav/navigators/baselines.py}
  \item Extended both \texttt{scripts/run\_benchmarks.py} and \texttt{scripts/run\_benchmarks\_3d.py} to include these baselines
\end{itemize}

\textbf{Legacy source anchors}

\begin{itemize}[leftmargin=*]
  \item \texttt{nodes/collision\_avoidance\_new.py}, branches \texttt{field == 3} and \texttt{field == 4}
  \item Cross-check against \texttt{nodes/collision\_avoidance\_okt.py}, which contains the same branch structure with slightly different hard-coded gains and cutoffs
\end{itemize}

\textbf{Porting notes}

\begin{itemize}[leftmargin=*]
  \item In the ROS repository these methods were implemented as obstacle-only terms added on top of the same go-to-goal node, so the clean Python port keeps the existing goal controller and swaps only the obstacle-response term.
  \item The old ROS snapshots do not fully agree on exact scalar gains, distance cutoffs, or whether an extra factor of \(10\) multiplies the Haddadin branch. The clean Python port therefore preserves the branch logic and geometry, while using stable gains tied to the current benchmark framework.
  \item The old ROS code used an \texttt{obs\_central} topic. In the clean Python port this is approximated by the centroid of the nearest active obstacle: exact center for circles and spheres, polygon centroid for 2D polygons, and footprint-centroid plus mid-height for prisms.
  \item The Chang gyroscopic branch from the ROS repository was \emph{not} ported in this change because the legacy implementation outputs only a planar angular velocity command for unicycle-like robots, not a fully actuated double-integrator acceleration.
\end{itemize}

\textbf{Implemented formulas}

For Haddadin, the obstacle term uses the sensed obstacle vector \(r_o\), a local obstacle-center proxy \(m_x\), goal vector \(r_g\), and robot velocity \(v\):

\begin{align}
\hat{l}_a &= \frac{v}{\lVert v \rVert}, \\
d &= \lVert r_o \rVert, \\
\hat{g} &= \frac{r_g}{\lVert r_g \rVert}, \\
d_v &= m_x - (m_x^\top \hat{g})\hat{g}, \\
\hat{t} &\propto \left(\frac{-r_o}{d}\right) \times \frac{d_v \times r_g}{\lVert d_v \times r_g \rVert}, \\
B &= \frac{\hat{t} \times v}{\lVert v \rVert d^2}, \\
F_{\text{Haddadin}} &= k_H \left(\hat{l}_a \times B\right)\lVert v \rVert .
\end{align}

For Sabattini, the obstacle term uses the goal vector \(r_g\), velocity \(v\), obstacle vector \(r_o\), and a damping term based on the signed approach speed:

\begin{align}
u_t &= 0.1\,r_g, \\
w &= u_t - \operatorname{proj}_{v}(u_t), \\
u_g &= -k_S \frac{w}{\lVert w \rVert}, \\
\sigma &=
\begin{cases}
1, & v^\top r_o \ge 0, \\
0, & v^\top r_o < 0,
\end{cases} \\
F_{\text{damp}} &= -0.05\,s\,\frac{E_0}{\lVert r_o \rVert}
\left(\lvert v^\top r_o \rvert + e^{-\lvert v^\top r_o \rvert}\right)
\frac{r_o}{\lVert r_o \rVert}, \\
F_{\text{Sabattini}} &= \sigma\,(u_g + F_{\text{damp}}),
\end{align}

where \(E_0 = 0.1\,d_{g,0}^2\) uses the initial goal distance and \(s \in \{+1,-1\}\) follows the sign convention from the ROS code.

\section*{Entry 2026-06-19: Added Differential-Drive Kinematic Model}

\textbf{What changed}

\begin{itemize}[leftmargin=*]
  \item Added \texttt{DifferentialDriveState} and \texttt{DifferentialDriveModel} in \texttt{src/mfinav/models/differential\_drive.py}
  \item Added \texttt{simulate\_differential\_drive} in \texttt{src/mfinav/sim/differential\_drive.py}
  \item Added 2D nonholonomic benchmark script \texttt{scripts/run\_benchmarks\_diff\_drive.py}
\end{itemize}

\textbf{Motivation}

\begin{itemize}[leftmargin=*]
  \item The repository previously benchmarked only fully actuated double-integrator motion.
  \item A differential-drive model is needed to test the same navigation algorithms on a nonholonomic planar robot while keeping the sensing and obstacle abstractions unchanged.
\end{itemize}

\textbf{Modeling choice}

The navigation algorithms in this repository output a planar guidance vector
\(u \in \mathbb{R}^2\). For the differential-drive robot, this vector is
interpreted as a desired heading and speed command rather than as a physical
acceleration.

Let the robot state be
\[
x = \begin{bmatrix} p_x & p_y & \theta & v & \omega \end{bmatrix}^\top
\]
with planar position \(p\), heading \(\theta\), linear speed \(v\), and angular
speed \(\omega\). Given guidance vector \(u\), define

\begin{align}
\theta_d &= \operatorname{atan2}(u_y, u_x), \\
e_\theta &= \operatorname{wrap}(\theta_d - \theta), \\
\omega_c &= \operatorname{sat}_{\omega_{\max}}(k_\theta e_\theta), \\
v_c &= \operatorname{sat}_{v_{\max}}(k_v \lVert u \rVert)\,\max(0,\cos e_\theta).
\end{align}

The kinematic propagation is then

\begin{align}
\dot{p}_x &= v_c \cos \theta, \\
\dot{p}_y &= v_c \sin \theta, \\
\dot{\theta} &= \omega_c.
\end{align}

\textbf{Interpretation}

\begin{itemize}[leftmargin=*]
  \item This is a tracking wrapper around the existing navigation field, not a re-derivation of the original papers for differential-drive actuation.
  \item The same local sensing and obstacle models are reused, so differences in benchmark behavior mainly reflect the effect of the nonholonomic kinematics and heading-tracking layer.
  \item Differential-drive benchmarks are therefore useful for qualitative portability studies and algorithm comparison, but should not be interpreted as a paper-faithful nonholonomic controller unless a dedicated derivation is added later.
\end{itemize}

\section*{Entry 2026-06-19: Added 3D Quadrotor Dynamics Model and Benchmark Path}

\textbf{What changed}

\begin{itemize}[leftmargin=*]
  \item Added \texttt{QuadrotorState} and \texttt{QuadrotorModel} in \texttt{src/mfinav/models/quadrotor.py}
  \item Added \texttt{simulate\_quadrotor} in \texttt{src/mfinav/sim/quadrotor.py}
  \item Exported the quadrotor model and simulator through the package compatibility layer
  \item Added dedicated 3D quadrotor benchmark script \texttt{scripts/run\_benchmarks\_quadrotor\_3d.py}
\end{itemize}

\textbf{Motivation}

\begin{itemize}[leftmargin=*]
  \item The clean Python repository already supported 3D point-mass navigation, but not the quadrotor dynamic model used in the ICRA 2019 ROS implementation.
  \item A dedicated quadrotor model is needed so the same 3D navigators can be tested under a more realistic underactuated vehicle assumption.
\end{itemize}

\textbf{Modeling choice}

The quadrotor state is
\[
x = \begin{bmatrix} p^\top & v^\top & R & \omega^\top & T \end{bmatrix},
\]
where \(p \in \mathbb{R}^3\) is position, \(v \in \mathbb{R}^3\) is linear
velocity, \(R \in SO(3)\) is attitude, \(\omega \in \mathbb{R}^3\) is angular
velocity, and \(T\) is total thrust.

The high-level navigation layer still outputs a 3D guidance vector
\(u \in \mathbb{R}^3\). In the quadrotor model this is interpreted as the
desired translational navigation force relative to hover:
\[
F_Q = u + m g e_3 .
\]

The rigid-body propagation follows the same structure as the legacy ROS
quadrotor node:
\begin{align}
\dot{p} &= v, \\
\dot{v} &= \frac{1}{m}\left(R e_3 T + G\right), \\
\dot{R} &= R \hat{\omega}, \\
\dot{\omega} &= J^{-1}\left(J \hat{\omega}\,\omega + \tau\right).
\end{align}

\textbf{Low-level actuation assumption}

To match the practical quadrotor controller path in the old ROS implementation,
the clean Python model uses a thrust plus attitude-tracking layer rather than
driving the navigation field directly into rotor speeds.

The total thrust tracks the desired force magnitude:
\[
T_d = \lVert F_Q \rVert .
\]

Desired roll and pitch are computed from the horizontal components of \(F_Q\)
and the available thrust, with yaw regulated toward zero:
\begin{align}
\phi_d &= -k_f \frac{F_{Q,y}}{\max(T_d,\varepsilon)}, \\
\theta_d &= k_f \frac{F_{Q,x}}{\max(T_d,\varepsilon)}, \\
\psi_d &= 0 .
\end{align}

The torque command is then a PD law on roll, pitch, and yaw:
\begin{align}
\tau_\phi &= -k_\phi (\phi - \phi_d) - d_\phi \omega_x, \\
\tau_\theta &= -k_\theta (\theta - \theta_d) - d_\theta \omega_y, \\
\tau_\psi &= -k_\psi (\psi - \psi_d) - d_\psi \omega_z .
\end{align}

Rotor speeds are computed through the Pelican-style allocation matrix used in
the old ROS code, but the internal benchmark dynamics are propagated with the
commanded thrust and torques, matching the legacy non-Gazebo simulation path.

\textbf{Current benchmark outcome}

The first quadrotor benchmark pass is now operational and writes:
\begin{itemize}[leftmargin=*]
  \item \texttt{artifacts/benchmark\_comparison\_quadrotor\_3d.png}
  \item \texttt{artifacts/benchmark\_metrics\_quadrotor\_3d.csv}
\end{itemize}

At the current tuning level:
\begin{itemize}[leftmargin=*]
  \item the quadrotor dynamics are stable enough to simulate all current 3D navigators across the default 3D benchmark set,
  \item the existing 3D double-integrator benchmark path still runs unchanged after the addition,
  \item the existing 2D benchmark path still runs unchanged after the addition,
  \item but the quadrotor benchmark results are not yet as strong as the point-mass 3D results and still need dedicated gain tuning, especially in the clustered and cavity-like environments.
\end{itemize}

\textbf{Interpretation}

\begin{itemize}[leftmargin=*]
  \item This addition creates the needed structural bridge from the clean 3D navigation layer to the older quadrotor paper and ROS implementation.
  \item The current quadrotor path is already useful for qualitative comparison and model-portability testing.
  \item Full paper-level reproduction of the quadrotor results will likely require a second tuning pass that is specific to the quadrotor actuation and benchmark geometry, rather than reusing the point-mass gains directly.
\end{itemize}
